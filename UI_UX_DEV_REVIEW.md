# v6 UI/UX 與開發審視紀錄

## UI/UX 結論

- 首頁只保留一張最大結果卡，避免使用者需要閱讀大量說明。
- 掃描流程改為逐項掃描：條碼、品名、效期/批號。
- 每個掃描項目只顯示已掃、請掃、未掃。
- 不合格命中後自動停止掃描，不再要求使用者補齊其他資料。
- 詳細來源與 OCR 記錄收在摺疊區，避免干擾一般使用者。

## 前端開發結論

- 保留 GitHub Pages 靜態網站架構，不需要後端。
- `data/recalls.json` 與根目錄 `recalls.json` 同時提供，降低部署路徑錯誤。
- Service worker 對資料檔採 network-first，抓不到新版時才使用快取。
- 每次開頁、重新整理或按「更新」都會嘗試抓最新版資料。

## QA 測試結論

- 條碼命中但缺效期：顯示需確認。
- 條碼 + 效期命中：顯示不合格。
- 品名 + 效期命中：顯示不合格。
- 無條碼商品可用品名 + 批號 + 效期判斷。
- 未收錄條碼顯示無資料，不顯示合格。

## 資料治理結論

- `source_urls` 建議逐筆產品保留。
- DEMO Safe 測試商品已移除，避免朋友誤解為真實合格資料。
- 正式使用時，應把官方與業者資料轉成同一個 JSON schema。
